# Note Taxonomy Design

Design logical folder hierarchies and naming conventions for knowledge vaults.

## When to Use

This skill activates when:

- User asks to organize or restructure their vault
- Discussion of folder hierarchy or naming conventions
- Creating new organizational schemes
- Improving discoverability of notes

## Workflow

### 1. Survey Existing Structure

- List current folders and root notes
- Identify patterns and content types
- Note any inconsistencies

### 2. Propose Organization

- Group by: project, topic, status, type, or time
- Define naming conventions (Title Case vs kebab-case)
- Create clear hierarchy levels

### 3. Document the Taxonomy

- Create a `Taxonomy.md` reference
- Include examples of each category
- Document naming rules

## Principles

1. **Make structure obvious** before adding more structure
2. **Preserve user language** unless clearer convention exists
3. **Keep notes discoverable** through titles, tags, and links

## Output Format

```markdown
## Proposed Taxonomy

Root/
├── 01-Projects/ # Active projects
├── 02-Reference/ # Reusable knowledge
├── 03-Archive/ # Completed/inactive
└── \_templates/ # Note templates

Naming: Title Case for notes, kebab-case for folders
```
